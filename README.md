# device-

## Purpose / Introduction

The purpose of this device is to create a simple STM32 device that is capable of running a timer and playing music.

## Installation

These are the steps required to replicate the environment used by myself:

1. Install and Open STM32CubeMX.
    1. Click on Project Manager
    2. Ensure Toolchain / IDE selected is CMake. Other options can be left as is
2. Install and Open Visual Studio Code (VS Code).
    1. On the left hand bar, click on Extensions (or Ctrl + Shift + x)
    2. Install 'STM32CubeIDE' for Visual Studio Code
    3. On the topbar, press File -> Open Folder (Ctrl + K, then Ctrl + O) and select the `Firmware` folder
3. Install Ruby for your system. This is used to automate the generation of the mock files from CMock.
4. In VS Code, press the Run and Debug button, and Run using (Ctrl + F5)
5. Once debugging mode has initialised, press the play button (F5) to begin running code
    1. Note this firmware has been downloaded, and so it can run from power alone

## Usage

Once the firmware is flashed and the device is powered on, it will initialise automatically. 

The UI Pages Follow the following Structure:

```
device-
└── Home
    ├── Music
    │   ├── All Songs
    │   ├── Playlists
    │   ├── Artists
    │   └── Albums
    ├── Timer
    │   ├── Preset Timers
    │   └── Set Timer
    └── Settings
        ├── Brightness
        └── Modes
```

### Music Playback

Music playback is triggered once the timer reaches its target, or can be activated independently depending on configuration.

### Countdown/Timer Functions

The timer can be started, stopped, and reset using the designated input method (e.g. onboard button). 

### Communication

Serial output via a UART connection can be monitored using a tool such as PuTTY or the VS Code serial monitor to observe device state and debug output.

## Coding Style

This project follows a consistent C coding style suitable for embedded development. The key conventions are:

- **Naming**: Functions and variables use `snake_case`. Macros and constants use `UPPER_SNAKE_CASE`. Structs and typedefs use `PascalCase`.
- **Indentation**: 4 spaces — no tabs.
- **Comments**: Public functions are documented with a brief description above the declaration. Inline comments are used sparingly and only where the intent is not immediately obvious.
- **File structure**: Each module has a paired `.c` and `.h` file. Headers use include guards (`#ifndef`/`#define`/`#endif`).
- **HAL usage**: STM32 HAL functions are preferred over direct register access unless performance requires otherwise.

## Test

Unit tests are written using [Unity](https://github.com/ThrowTheSwitch/Unity) with mocks generated by [CMock](https://github.com/ThrowTheSwitch/CMock). Ruby is required to run the mock generation scripts (see Installation).

To run the tests:

1. Navigate to the `test/` directory.
2. Run the mock generation script: `ruby generate_mocks.rb` (or equivalent).
3. Build and run the test suite via CMake:

```bash
cmake -S . -B build/test -DTARGET=test
cmake --build build/test
./build/test/run_tests
```

Tests are organised per module and cover core logic independent of hardware. Hardware-dependent behaviour should be validated on-device.

## Maintenance

This project is maintained on a best-effort basis. To keep the project healthy:

- **Dependencies**: STM32CubeMX should be used to regenerate HAL/middleware files if updating the STM32 firmware package. Avoid manually editing auto-generated files.
- **Updating the project**: After pulling changes, re-run CMake configuration to pick up any changes to `CMakeLists.txt`.
- **Toolchain**: If updating the ARM GCC toolchain, verify the version is compatible with the CMake toolchain file (`cmake/gcc-arm-none-eabi.cmake` or equivalent).
- **CMock/Unity**: If updating Unity or CMock, re-run mock generation to ensure test stubs are in sync with updated headers.
